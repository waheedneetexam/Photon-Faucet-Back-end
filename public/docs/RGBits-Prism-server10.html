⚡ RGBits Prism Server v10

This page documents what Prism Server does today (regtest/dev): authentication, virtual wallets, BTC Lightning payments, RGB asset support, Prism split engine, webhooks, and websocket events.

Repo RGBits-Prism-server

Service rgbits-prism-api (Axum, Rust)

Dependencies Postgres 16, Redis 7, RGB Lightning Node (RLN) over HTTP

1. What Prism Server Is

Prism Server is the backend API that turns “wallet actions” into node actions. It is the place where we keep the truth for: users, wallet identities, payment history, and (in regtest/dev) simplified BTC and RGB balances.

  • It talks to RLN using RLN_BASE_URL + RLN_AUTH and calls endpoints like lninvoice, sendpayment, decodelninvoice, listpayments.
  • It stores state in Postgres (wallets, assets, balances, payments, prism rules, executions, webhooks).
  • It pushes events using Redis pub/sub (wallet scoped) and can also deliver events to HTTP webhooks.

3. Authentication Model

Auth is challenge-response (secp256k1) + JWT.

  • POST /auth/enroll: enroll a pubkey + bitcoin address into users.
  • GET /auth/challenge: returns a nonce challenge for that pubkey.
  • POST /auth/verify: verifies ECDSA signature, returns access JWT + refresh token (refresh stored in Redis).
  • POST /auth/refresh and POST /auth/logout.
  • Authorization: Bearer <jwt> protects user endpoints.

Note: legacy users.email + password_hash may still exist in schema, but current auth flow is pubkey based.

4. Wallet Identity (Wallet Key Header)

Many endpoints require a logical wallet identity header: x-photon-wallet-key. This is the wallet’s “API identity”, not the mnemonic.

  • Payments and invoices are scoped to a wallet row in wallets.
  • One user can own multiple wallets (virtual wallets).

5. BTC Lightning Payments (Current Scope)

  • POST /payments/invoices creates a BOLT11 invoice (and stores a pending inbound row in payments).
  • POST /payments/pay pays a BOLT11 invoice (stores outbound pending, then settles/failed).
  • POST /payments/keysend sends keysend payments.
  • Reconciler loop polls RLN listpayments and updates pending payment rows to settled or failed.

BTC balance tracking in Prism Server is simplified for dev: it uses wallets.btc_balance_msat.

6. RGB Assets (Registry, Invoices, Payments, Issuance)

  • GET /assets, GET /assets/:asset_id read from rgb_assets.
  • On startup, Prism performs a best-effort RLN asset sync into rgb_assets.
  • POST /payments/invoices supports RGB invoices by including asset_id + asset_amount.
  • POST /payments/pay detects RGB invoices via decode fields and (dev model) checks/deducts spendable from rgb_balances.
  • On settlement (inbound, non-prism), Prism credits the receiver wallet rgb_balances.
  • POST /assets/issue is admin-only issuance (calls RLN, upserts rgb_assets, credits issuer supply).

7. Prism Split Engine

Prism rules define how an inbound invoice should be split across recipient wallets.

  • Rule CRUD: POST/GET/PUT/DELETE /prism/rules
  • Invoice generation: POST /prism/rules/:id/invoice creates an invoice linked to payments.prism_rule_id.
  • On settlement: Prism reads the rule, calculates split amounts (floor division), remainder goes to the “owner” split entry, credits recipients in one DB transaction, and writes an audit row in prism_executions.
  • Audit: GET /prism/rules/:id/splits returns historical executions.
  • External targets (PRISM-06): a split item can include external_invoice (BTC only, paid best-effort after commit).

Percent mode is currently the main supported mode: sum of share_pct must equal 100.

8. Webhooks (WH-01..WH-03)

  • Create/list/delete: POST /webhooks, GET /webhooks, DELETE /webhooks/:id.
  • Deliveries are queued in webhook_deliveries with status and retry timing.
  • Dispatcher loop posts JSON to webhook URLs.
  • Signature header: X-RGBits-Signature = HMAC-SHA256 hex over the raw JSON payload.
  • Retries: backoff schedule up to about 72 hours.

Secret storage currently uses webhooks.secret_hash as a secret field (not hashed). This is acceptable only for dev; production should encrypt-at-rest or store a hash and do per-request derived signatures.

9. WebSocket Events (WH-04)

Wallet-scoped websocket stream:

  • GET /ws?token=<jwt>&wallet_id=<wallet_uuid>
  • Internally subscribes to Redis pubsub channel: wallet:<wallet_id>:events
// Browser example (dev)
const ws = new WebSocket("wss://YOUR_HOST/ws?token=JWT&wallet_id=WALLET_UUID");
ws.onmessage = (ev) => console.log("event", ev.data);

10. Environment Variables

PRISM_BIND
Bind address, e.g. 0.0.0.0:4000
DATABASE_URL
Postgres URL
REDIS_URL
Redis URL
JWT_SECRET
JWT signing secret
RLN_BASE_URL
RLN HTTP base URL
RLN_AUTH
Authorization header value for RLN
PAYMENT_RECONCILE_INTERVAL_MS
Poll interval for settlement reconciler
