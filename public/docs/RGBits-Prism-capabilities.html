⚡ RGBits Prism Server Capabilities

This page explains what RGBits Prism Server can do today, what it cannot do yet, and how it fits into channels, invoices, RGB assets, and UTXO handling.

Naming Fix In the UI and docs we will call the “LNbits API Key” the RGBits Prism Server API Key. It is the same thing in our setup.

Why EXT-02 exposes LNbits-compatible endpoints, and they use X-Api-Key. That key is generated by Prism Server.

1. What Prism Server Is

Prism Server is your backend control plane. It does not replace Bitcoin. It orchestrates:

  • Authentication and multi-tenant wallet identities.
  • BTC Lightning invoices and payments via RLN.
  • RGB assets registry and RGB Lightning invoices/payments via RLN.
  • Prism “split rules” (one inbound payment, many recipients).
  • Operational admin actions (user ops, channels, fee policy).
  • Webhooks and websocket events for apps that need real-time updates.

3. Invoices and Payments

BTC

  • Create invoice: POST /payments/invoices (amt_msat)
  • Pay invoice: POST /payments/pay (invoice)
  • Keysend: POST /payments/keysend
  • Reconciliation: background loop polls RLN listpayments and updates DB status.

Balance is dev-mode: wallets.btc_balance_msat is not a real node balance sync yet.

RGB

  • Registry: GET /assets, GET /assets/:asset_id
  • Create RGB invoice: POST /payments/invoices with asset_id, asset_amount
  • Pay RGB invoice: POST /payments/pay (deducts spendable in dev model)
  • Settlement credit: inbound RGB non-prism gets credited into rgb_balances on “settled”.

RGB non-Lightning transfers are not implemented in Prism today.

4. Channels (What They Are, Who Creates Them)

A Lightning channel is a funded two-party connection. You lock BTC into a 2-of-2 output, then payments move instantly off-chain.

  • Prism can manage channels via admin endpoints: GET /admin/channels, POST /admin/channels/open, close, rebalance.
  • Users do not self-open asset channels in product flow yet. That’s an infrastructure step because it needs BTC funding and operational safety.

5. UTXO Handling (The Big Question)

You are correct: if Prism Server starts managing UTXOs for many users, it becomes an operational problem. But it is solvable if we treat it as a tenant-scoped ledger with strict rules.

Today Prism does not implement a full deposit + redeem UTXO vault.

Future Model (tenant-based UTXO custody for regtest/dev, or carefully in mainnet):

  1. Per-user deposit address: Prism gives User 1 a unique BTC address.
  2. Detect deposit: backend watches Bitcoin (electrum/bitcoind/electrs) and detects UTXOs to that address.
  3. Assign ownership: store UTXO row with user_id, wallet_id, outpoint (txid:vout), value, confirmations, spend_state.
  4. Reserve for operations: when issuing assets, opening channels, or on-chain sends, Prism reserves specific UTXOs.
  5. Redeem: user requests withdrawal; Prism builds and broadcasts a transaction spending only that user’s UTXOs.

This requires strict controls: UTXO slot management, fee estimation, dust rules, consolidation strategy, and audit logs.

6. “Will It Be A Headache?” Answer

It becomes a headache if we do not build guardrails. The guardrails are:

  • UTXO slots: fixed number of “issuance-ready” UTXOs per user (prevents unlimited fragmentation).
  • Consolidation policy: periodically merge small UTXOs into fewer bigger ones (with user attribution preserved).
  • Strict accounting: every reservation/spend is recorded and reconciled against chain.
  • Tenant isolation: never mix user UTXOs in the same spend unless you intentionally implement coinjoin-like batching (not recommended for v1).

7. What You Can Integrate In The Chrome Extension Right Now

BTC LNbits-compatible
