RGB Issuance to Primary Channel Bootstrap

This document defines the product and backend workflow for issuing a new RGB asset and, optionally, bootstrapping the first RGB Lightning channel for that asset. It is the design contract for board task PHO-029.

Registry-onlyAn asset exists, is listed in Asset Registry, and can be imported, but has no guaranteed Lightning path yet.
Lightning-readyAn asset exists, has funded channel liquidity, and has at least one usable primary RGB Lightning channel.
Primary channelThe first RGB Lightning channel opened for a new asset so the asset can move off-chain immediately after issuance.

1. Problem Statement

Today, RGB asset issuance and RGB Lightning liquidity are separate. A user can issue an asset and have it appear in Asset Registry while the same asset is still unusable for Lightning because no funded channel exists yet.

That creates product ambiguity. Users see a newly issued asset and assume instant transfer is available, but the asset may still be in a registry-only state. The missing piece is a clear workflow that links issuance, liquidity reservation, BTC channel funding, channel opening, and post-issuance readiness status.

Registry visibility and Lightning usability must be treated as separate states. An asset can be real and public without being Lightning-ready.

2. Target User Flow

  1. User opens Issue Asset in the wallet.
  2. User enters name, ticker, precision, and total supply.
  3. User chooses whether to bootstrap Lightning liquidity now.
  4. If yes, user chooses:
    • percentage of issued supply reserved for channel liquidity
    • BTC amount to commit to the channel
    • whether BTC funding happens during issuance or after issuance
  5. Backend issues the asset.
  6. Backend stores issuance state and liquidity plan.
  7. Backend lists the asset in Asset Registry.
  8. If bootstrap was requested and prerequisites are met, backend creates the first RGB Lightning channel.
  9. Wallet shows the asset as either Registry-only or Lightning-ready.

3. States and Definitions

BOOTSTRAP_FAILEDIssuance succeeded, but channel bootstrap failed or only partially completed.Asset exists, but Lightning setup needs retry or manual recovery.

4. Core Product Rules

  • The full issued supply is minted into the requesting user wallet first.
  • Any portion reserved for Lightning liquidity is a separate post-issuance reservation, not a hidden issuance diversion.
  • Asset Registry listing should happen as soon as issuance succeeds, even if Lightning bootstrap is still pending.
  • The wallet must clearly show that registry listing does not automatically mean Lightning readiness.
  • Primary channel bootstrap is optional. A user must be able to issue an asset without opening a channel immediately.
  • BTC funding and RGB liquidity planning must be validated before bootstrap begins.

5. Liquidity Planning Rules

Reserved RGB Amount

The issuer can choose a percentage of total supply to reserve for channel liquidity. The backend converts that percentage into an exact reserved asset amount.

  • If total supply is 1,000,000 and liquidity percentage is 10%, reserve 100,000 units.
  • The remainder stays in the issuer wallet as regular user-owned balance.

BTC Channel Funding

RGB Lightning channels still require Bitcoin funding. The user must choose or confirm the BTC amount to commit for the primary channel.

  • Fund now: BTC is supplied during issuance workflow.
  • Fund later: issuance completes first and the channel stays in waiting state until BTC arrives.
The exact minimum BTC requirement is a backend-calculated readiness value and must be shown in the wallet before final confirmation.

6. Backend Workflow Contract

1. Validate issuance request 2. Validate optional channel bootstrap settings 3. Issue RGB asset into requesting user wallet 4. Record issuance row with liquidity plan and lifecycle status 5. Insert/update Asset Registry row 6. If bootstrap not requested: mark ISSUED_REGISTRY_ONLY 7. If bootstrap requested but BTC missing: mark WAITING_BTC_CHANNEL_FUNDING 8. If bootstrap requested and ready: a. reserve configured RGB amount b. allocate BTC channel funding c. open primary RGB Lightning channel d. verify channel ready/usable e. mark LIGHTNING_READY 9. On failure after issuance: mark BOOTSTRAP_FAILED and preserve full audit trail

7. Failure Handling Rules

  • If issuance fails, nothing is registered.
  • If issuance succeeds and registry insertion succeeds, the asset remains valid even if channel creation fails later.
  • If channel bootstrap fails, do not delete the asset. Keep it registry-visible and show BOOTSTRAP_FAILED.
  • Reserved liquidity and BTC funding state must remain auditable for retry or manual repair.
  • The wallet must not display Lightning-ready until the channel is confirmed ready and usable.

8. Wallet Requirements

  • Add a Bootstrap Lightning liquidity toggle to the issuance form.
  • Add a percentage field or slider for reserved RGB liquidity.
  • Add BTC channel funding amount input with backend readiness hint.
  • Support fund now and fund later modes.
  • Show the post-issuance state:
    • registry-only
    • waiting for BTC funding
    • bootstrapping
    • Lightning-ready
    • bootstrap failed

9. Data Model Changes

The issuance workflow table should persist at least:

  • liquidity_percentage
  • reserved_asset_amount
  • requested_channel_btc_sat
  • channel_bootstrap_mode (none, fund_now, fund_later)
  • lifecycle_status
  • primary_channel_id if created
  • last_error for retry and support visibility

10. Implementation Sequence

  1. Persist liquidity plan and lifecycle status.
  2. Add wallet issuance controls.
  3. Add BTC funding readiness and mode selection.
  4. Implement primary RGB Lightning channel bootstrap orchestration.
  5. Add post-issuance readiness UI.
